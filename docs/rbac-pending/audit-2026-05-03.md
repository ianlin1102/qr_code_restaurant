# RBAC 审计 — 待修问题清单

**日期**: 2026-05-03
**审计范围**: 员工管理 + 角色管理 + 权限管理 (admin 端)
**结论**: 框架对了 (~60%), 细节没打磨好. 8 类问题 + 3 优先级行动清单.

---

## TL;DR

3 层 RBAC 架构 (module licensing × role × user) 是 SaaS 标准, 本身合理. 但:

- Manager 默认**不能管员工** (不含 `staff:manage`) — 命名与功能严重错位
- Permission 粒度太粗 (`orders:write` 同时含 void/refund/转单, waiter 可作废订单 = 风险)
- 角色覆盖不全 — 只 owner/manager/waiter, 缺餐饮业必备 (kitchen / cashier / host / shift-lead)
- UI 18 个权限不分组, 切换 role 无 confirm
- legacy `role: string` 双轨制未收尾
- 缺 audit log (合规风险)

---

## 架构总览

**3 层 RBAC**:

1. **Module licensing** (店铺级) — `MODULE_REGISTRY` 定义 6 module:
   - `core` (required, 10 perms: orders/menu/tables/settings/billing × R/W)
   - `analytics` / `coupons` / `waitlist` / `staff-management` / `printer`
   - `moduleLicense.modules` 决定店铺能用什么权限池
2. **Role definition** (店铺级) — 3 系统角色 (owner/manager/waiter) + 自定义角色, 每角色含 18 个 perm 子集
3. **User assignment** — `StoreUser.role` (legacy string) 或 `roleId` (新)

**最终权限计算**:
```
final_perms = role.permissions ∩ store.modulePermissions
```

**关键文件**:
- `shared/types.ts` — Permission union (18 个) + `RoleDefinition` / `StoreUser` interface
- `shared/modules.ts` — `MODULE_REGISTRY`
- `server/src/controllers/role.service.ts` — system role 定义 + `resolvePermissions`
- `server/src/lib/module-permissions.ts` — module licensing intersect
- `client/src/pages/admin/StaffManagePage.tsx` — UI

## ✅ 做对的地方

| 设计 | 评 |
|---|---|
| 多层 RBAC (module × role × user) | 标准 SaaS multi-tenant pattern |
| `resource:action` 命名 (e.g. `orders:write`) | OAuth2 scope-style, 清晰 |
| `staff:manage` 与 module licensing intersect | 没买就强制无 — 商业硬约束 |
| Owner 不可改不可删 | 防止把自己锁出 |
| System role 不可删 | 防误删致系统不可用 |
| Role store-scoped | Multi-tenant 隔离正确 |
| ClockPin 4-digit PIN 打卡 | 不用每次输密码, 实务合理 |

## ⚠️ 问题清单 (8 类)

### 1. Manager 默认 NO `staff:manage` — 命名与功能严重错位 🚨

**现状** (`server/src/controllers/role.service.ts:30-31`):
```ts
const MANAGER_PERMISSIONS = ALL_PERMISSIONS.filter(p => p !== 'staff:manage')
```

**问题**: 真实餐馆的 Manager **就是管员工** 的人 (排班 / 加班 / 招新 / clock-in 监督). 现在 Manager 反而**不能管员工**, 老板必须亲自处理新员工入职. 不 scalable, 也违反业界共识.

**建议**: Manager 默认**应含** `staff:manage`. 或改名 'shift-lead' 表示"当班负责人"避免歧义.

---

### 2. Permission 粒度太粗 / 太细 🚨

| Perm | 问题 | 建议 |
|---|---|---|
| `orders:write` | 改单 / void / refund / 转单 全在一起. waiter 能 void 单 = 风险 | 拆 `orders:void` + `orders:refund` (manager 才有) |
| `staff:manage` | 看员工 vs 改密码 vs 排班 都同权限 | 拆 `staff:read` / `staff:write` / `staff:schedule` |
| `billing:read/write` | 看账单 vs 看财务报表 vs 改支付模糊 | 改名 `payments:*` + `accounts:*` 分开 |
| `printer:read` | 几乎用不上 (谁要"看 printer config 不改"?) | 合并 `printer:use` |

---

### 3. 角色覆盖不全 — 缺餐饮业必备角色 ⚠️

**业界标准餐饮 POS 角色**:
- Owner / GM
- Manager / Asst Manager
- **Shift Lead** ← 缺
- Server / Waiter
- **Bartender** ← 缺
- **Host / Hostess** ← 缺
- **Busser** ← 缺
- **Kitchen / Line Cook / Head Chef** ← 缺 (后厨权限典型: `orders:read` + `printer:write` + `tables:read`)
- **Cashier** ← 缺
- **Dishwasher** ← 缺

当前只有 owner/manager/waiter — **过于抽象**. 用户能自建 custom role 但**无 templates**, 一头雾水.

**建议**: seed 6-8 默认 role templates (kitchen-staff, cashier, host, bartender, busser, shift-lead) 给用户复制改.

---

### 4. UI / UX 问题 (StaffManagePage) ⚠️

| 问题 | 位置 | 建议 |
|---|---|---|
| 18 个 checkbox 散乱无分组 | Role dialog `ALL_PERMISSIONS.map` (line 386) | 按 category 分组 (Orders / Menu / Tables / Settings / Modules) |
| 创建新 role 起点 `['orders:read', 'menu:read']` 太低 | line 131 `openNewRole` | 加 "Copy from waiter / manager" preset 按钮 |
| Role 卡片列出 17 个 badge 占满屏 | line 302 | 折叠 "17 permissions, click to expand" |
| 切换 role inline Select 无 confirm | line 237 `handleRoleChange` | 添加确认弹窗, 防误点 lock out |
| Module unlicensed perm 不提示失效 | role dialog | checkbox 旁标 "(Module not licensed — inactive)" |

---

### 5. legacy `role: string` vs `roleId` 双轨制未收尾 ⚠️

**现状** (`shared/types.ts:61-62`):
```ts
role: string              // legacy
roleId?: string           // new
```

`resolvePermissions` 双分支处理 (`role.service.ts:130-155`). 增加复杂度.

**建议**: 写迁移脚本把 user.role → roleId, 弃 legacy field.

---

### 6. 缺审计 / Audit Log 🚨

- 谁删了员工? 谁改了 role 权限? 谁给 manager 加了 staff:manage?
- 当前只 `logger.info` → console, **不存 db / 无 UI 查询**.
- 商业 SaaS 必备 (合规 / 安全调查).

**建议**: 新增 `audit_log` table + 写中间件捕捉 staff/role write 操作 + 简单 UI 查询.

---

### 7. `canDelete` server-side 缺保护 ⚠️

Frontend `canDelete` 检查 owner 自删 + 最后 1 个 owner. 但**没在 server 端强制 "owner 只能由 owner 删"**.

假设有人给 manager 加了 `staff:manage` (虽然现在 manager 默认没有), manager 能删 owner. 应在 `deleteStaff` controller 加 actor role 检查.

---

### 8. `resolvePermissions` legacy 分支硬编码 ⚠️

**现状** (`role.service.ts:144-149`):
```ts
} else if (legacyRole === 'owner') {
  rolePerms = ALL_PERMISSIONS
} else if (legacyRole === 'manager') {
  rolePerms = MANAGER_PERMISSIONS
} else if (legacyRole === 'staff' || legacyRole === 'waiter') {
  rolePerms = WAITER_PERMISSIONS
}
```

如果 system role 在 db 里被改 (虽然现在 owner 不能改), legacy fallback 仍用旧 ALL_PERMISSIONS — **不一致**. 应统一从 `roleStore` 查.
